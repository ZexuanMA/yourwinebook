# Your Wine Book — Codex MVP 总控执行清单

> 目标：保留现有 Web UI 与主要 API 形态不变，完成数据底座统一、移动端 MVP 两条核心闭环、上线前自检与回退准备。
>
> 使用方式：这不是提案文档，而是执行文档。后续排期、开发、验收、自检，都以本文件为单一准绳。

---

## 1. 成功定义

本次 MVP 只有四个真正的成功标准：

1. **统一数据与认证底座**
   - Web 与 App 必须共用同一套 Supabase 数据源与认证体系。
   - 不允许继续维持“Web 读 JSON / App 读 Supabase”的双轨状态。

2. **跑通两个核心闭环**
   - 闭环 A：打开 App -> 授权定位 -> 附近门店 -> 收藏 -> 外部导航。
   - 闭环 B：浏览 Feed -> 发图文帖 -> 点赞/评论 -> 举报/拉黑 -> 回访。

3. **不破坏现有 Web 前台与后台**
   - 已有 Web 页面、组件、i18n、样式、路由结构尽量不改。
   - 重点改“底层数据源”和“认证层”，不是重写前端。

4. **具备上线前最低稳定性**
   - 有 RLS、上传校验、风控、审核、埋点、监控、回退开关。
   - 有明确验收清单、首发前检查清单、上线后观察指标。

## 2. 范围锁定

### 2.1 必做范围

- Web 端数据层与认证层迁移到 Supabase。
- Expo 移动端 App 基础骨架。
- 找店闭环：定位、手动选区、附近门店、收藏、导航。
- 社区闭环：Feed、发帖、点赞、评论、收藏、举报、拉黑。
- Web B 端门店维护、官方发帖、审核后台。
- 安全上传链路、种子数据导入、基础埋点与监控。

### 2.2 明确不做

- 电商 / 支付 / 购物车
- 站内 IM
- 推荐算法
- 瀑布流
- App 内路线规划
- 短视频
- 话题热榜
- KOL 认证
- UGC 自动翻译

### 2.3 当前判断

- 现有 Web 的 UI / 页面层资产完整，原则上不动。
- `queries.ts` 的 Supabase 查询分支已具备较高复用价值，应优先复用，而不是推倒重写。
- 真正高风险区在于：
  - `mock-auth.ts` / `middleware.ts`
  - `user-store.ts`
  - `merchant-store.ts`
  - `admin-store.ts`
  - `community-store.ts`
  - `application-store.ts`
  - `price-store.ts`
- `community-store.ts` 是整个迁移的最高风险点，因为原模型是 JSON 嵌套结构，迁移后是多张关系表，需要适配层。
- `analytics-store.ts` MVP 阶段可以保留 SQLite，产品行为分析改由 PostHog 承担，不强行迁移到 Supabase。

## 3. 开发逻辑

整个项目必须按下面的顺序推进，不能倒序：

1. **先稳住工程骨架**
   - Monorepo、Expo、共享包、类型包、Schema、RLS 是一切后续开发的地基。

2. **再打通统一底座**
   - 上传、审核、种子数据、Auth、Store 迁移，决定 Web 和 App 是否真正共享一套数据。

3. **再做业务闭环**
   - 先做“找店”，因为它链路短、验证价值直接、对种子门店数据依赖高。
   - 再做“社区”，因为它链路长、上传和审核依赖更重、风控要求更高。

4. **最后做加固和灰度**
   - 草稿箱、弱网恢复、频率限制、清理任务、灰度分发、埋点校验、性能基线、合规文本，必须放在闭环跑通后做。

5. **每个阶段必须有硬门槛**
   - 不是任务做完就算结束，而是阶段退出条件全部满足，才能进入下一阶段。

## 4. 执行铁律

### 4.1 任务颗粒

- 以“一个可验证任务单元”为最小执行颗粒。
- 不允许跨阶段跳做。
- 不允许把多个高风险改动捆成一次大提交。

### 4.2 先读再改

- 每个任务开始前，先确认相关文件、依赖关系、导出函数、环境变量、现有脚本。
- 不在不了解上下文的前提下直接写代码。

### 4.3 保持外部接口稳定

- 迁移 Web store 时，**导出函数签名尽量不变**。
- 能通过内部适配层解决的问题，不要扩散到页面和 API route。
- Web 迁移的核心不是“写得更漂亮”，而是“无感换底座”。

### 4.4 Do No Harm

- 已有 Web 的页面、组件、样式、i18n、路由结构尽量不动。
- 只在必要处调整调用逻辑或兼容层。

### 4.5 遇错即停

- 任何 `build`、`typecheck`、`db reset`、集成测试失败，必须立即修。
- 不允许带着红灯进入下一任务。
- 不允许用注释掉报错、跳过脚本、临时删类型等方式假装通过。

### 4.6 所有命令非交互

- 初始化、安装、迁移、构建、测试都使用非交互命令。
- 禁止把工作流卡死在等待输入的步骤上。

### 4.7 进度必须外显

- 根目录维护 `AI_PROGRESS.md`。
- 每完成一个任务，记录：
  - 任务编号
  - 完成时间
  - 关键决策
  - 自检命令
  - 是否存在遗留风险

### 4.8 高风险任务必须留回退路径

- Auth、Store 迁移必须保留按模块切换的 feature flag。
- 推荐保留 `USE_SUPABASE_AUTH`、`USE_SUPABASE_[MODULE]` 类环境变量。
- 回退应允许单模块降级，而不是整体回滚。

### 4.9 阶段不闭环，不进入下一阶段

- 阶段目标未验收前，不追加 Phase 2 范围，也不引入“顺手做一下”的新需求。

### 4.10 标准开发与部署工作流（2026-03-17 起生效）

这一套工作流不是建议，而是默认规范。后续开发、部署、排障，统一按这套执行。

#### 运行架构

```text
浏览器 -> nginx:443 -> PM2 wine-prod -> next start -> port 3000（生产）
开发者 -> localhost:3001 -> next dev -> port 3001（开发）
```

核心原则只有一句话：

- **生产与开发端口彻底隔离**。
- 开发永远走 `3001`。
- 生产永远由 PM2 管理在 `3000`。
- 开发时**不需要**停生产服务。

#### 标准工作流

日常开发：

```bash
npm run dev
```

- 固定在 `localhost:3001` 预览。
- 外部网站继续由 PM2 在 `3000` 提供服务。
- 开发与生产互不干扰。

手动部署：

```bash
npm run deploy
```

- 等价于：`npm run build` + `pm2 restart wine-prod`

拉取代码后自动部署：

```bash
git pull
```

- 由 `.git/hooks/post-merge` 自动执行 build + PM2 restart。

#### 必须保留的运维不变量

即使后续 Phase 0a 做成 Monorepo，这几个规则也不能退化：

- 开发预览必须仍然走 `3001`，不能抢占生产端口。
- 生产进程必须仍然由 PM2 托管，不能直接手起 `next start`。
- 必须保留统一部署入口，优先是 `npm run deploy` 或其等价包装命令。
- `git pull` 后自动构建并重启生产服务的能力必须保留。

如果目录结构变化，比如未来改成 `apps/web`，也要通过根脚本把行为保持一致，而不是要求人手动记忆新流程。

#### 禁止事项

- **不要** `npm run start`
- **不要** `npx next start`
- **不要** 让 dev server 跑在 `3000`
- **不要** `pm2 delete wine-prod`

原因：

- `npm run start` / `npx next start` 会启动不受 PM2 管理的 Next 进程，最容易留下僵尸进程占住 `3000`。
- dev 跑到 `3000` 会和生产冲突。
- `pm2 delete wine-prod` 会直接破坏生产托管关系；只允许 `stop` / `restart`。

#### 旧经验的优先级说明

早期排障里曾出现“开发前先 `pm2 stop wine-prod`”的做法，那是**旧故障场景下的临时规避手段**，不是现在的标准流程。

从 2026-03-17 起，默认流程是：

- 开发时直接 `npm run dev`
- 不停 PM2
- 只在明确排查端口异常时，才介入 PM2 或端口占用诊断

### 4.11 高频故障排查速记

#### 场景 A：网站打不开，怀疑是 3000 端口被占

典型症状：

- `pm2 status` 里 `wine-prod` 进入 `errored`
- `pm2 logs wine-prod` 反复报 `EADDRINUSE`

标准处理步骤：

```bash
pm2 status
pm2 logs wine-prod
ss -tlnp | grep 3000
kill <PID>
pm2 restart wine-prod
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/zh-HK
```

判断标准：

- `ss -tlnp | grep 3000` 不应再看到异常残留的 `next-server`
- `pm2 restart wine-prod` 后服务恢复
- `curl` 返回 `200`

根因说明：

- 这类问题几乎都来自**不受 PM2 管理的 Next 进程**占住了 `3000`
- 最常见诱因就是直接跑了 `npm run start` / `npx next start`

#### 场景 B：首页或 Server Component 报 React / i18n 异常

如果组件是 `async` Server Component，就不要使用同步 hook：

- 不要在 async Server Component 中用 `useTranslations()`
- 不要在 async Server Component 中用 `useLocale()`

正确规则：

- async Server Component：用 `getTranslations()`、`getLocale()`，来源是 `next-intl/server`
- Client Component：才使用 `useTranslations()`、`useLocale()`

简化记忆：

- `async` + Server Component -> `await getTranslations()` / `await getLocale()`
- Client Component -> `useTranslations()` / `useLocale()`

#### 排障优先顺序

以后遇到“网站挂了/打不开/页面 500”，统一按下面顺序排：

1. 先看 PM2 是否正常。
2. 再看 `3000` 是否被异常进程占用。
3. 再看最近是否有人直接跑过 `next start`。
4. 最后再看应用层代码报错，例如 async Server Component 误用同步 hook。

## 5. 通用自检逻辑

### 5.1 每个任务都遵循同一个闭环

1. **Read**
   - 阅读涉及文件、配置、依赖、上下游调用点。
2. **Plan**
   - 明确本任务的边界、影响面、前置条件、回退方式。
3. **Act**
   - 只做当前任务定义内的代码和配置修改。
4. **Verify**
   - 运行本任务规定的自检命令。
5. **Smoke**
   - 至少做一次最小人工链路验证。
6. **Log**
   - 将结果写入 `AI_PROGRESS.md`。
7. **Gate**
   - 只有在自检成功、无阻塞报错时，才进入下一任务。

### 5.2 自检层级

#### L0. 文件级

- 文件创建成功
- 路径正确
- 配置存在

#### L1. 静态级

- `lint`
- `tsc --noEmit`
- `build`

#### L2. 数据级

- `supabase db reset`
- Schema 是否能完整建出
- RLS 是否按预期拦截
- 关键 SQL / RPC 是否可执行

#### L3. 集成级

- API route / store / Edge Function 互通
- Feature flag 切换有效
- 迁移后的返回格式兼容现有页面

#### L4. 冒烟级

- 最短业务路径手工验证一遍
- 错误分支至少验证一条

#### L5. 可观测级

- PostHog 事件可见
- Sentry 能捕获到错误
- 关键日志与告警规则存在

### 5.3 完成定义（Definition of Done）

一个任务只有同时满足下面条件，才可标记为完成：

- 任务要求的代码/配置/脚本已经落地。
- 自检命令全部通过。
- 影响到的关键页面或接口冒烟通过。
- 没有新增未解释的构建报错或类型错误。
- 已记录进度、关键决策、遗留风险。
- 有明确回退或降级策略（若该任务属于高风险改动）。

### 5.4 建议的进度记录格式

```md
# AI_PROGRESS.md

- [x] P0a-01 初始化 Monorepo
  - 决策：Web 迁入 apps/web，根目录只保留 workspace 配置
  - 自检：pnpm install / pnpm --filter web build
  - 风险：待验证 mobile 与 shared package 的路径解析

- [x] P0b-13 Web 认证层迁移
  - 决策：保留 legacy auth 文件，增加 USE_SUPABASE_AUTH 切换
  - 自检：pnpm --filter web tsc --noEmit
  - 风险：需继续验证 middleware 的角色判断
```

### 5.5 阻塞处理规则

- 如果是实现错误：当场修复。
- 如果是需求冲突：在 `AI_PROGRESS.md` 记录冲突点与建议处理方案。
- 如果是外部依赖阻塞：先完成可并行任务，不让整体排期停摆。
- 如果是数据库模型缺口：先补 schema，再继续业务开发。

- 种子数据已导入并可被业务消费。
- Web 认证层与关键 store 已迁移。
- fallback 与集成测试都已验证。

### 8.3 P1A 闭环验收

- 授权定位后 3 秒内看到附近门店。
- 拒绝定位后可手动选区继续使用。
- 门店列表含名称、地址、距离、营业状态。

- B 端可独立编辑门店信息、营业时间、坐标。

### 8.4 P1B 闭环验收

- 可发纯文字或最多 9 图帖子。
- 发帖失败时内容可恢复。
- Feed 时间倒序、游标分页正确。
- 点赞与评论即时可见，并能处理弱网失败。
- 举报支持帖子、评论、用户三类。
- 拉黑后 Feed 与评论区过滤生效。
- 管理员隐藏内容后 C 端不可见。

- 冷启动 TTI：<= 3s（4G 参考）

## 9. 风险与回退策略

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| RLS 复杂度超预期 | 高 | 底座延期 | 先做最小安全边界，复杂写操作放 Edge Function |
| Web -> Supabase 迁移引发回归 | 高 | 现有 Web 受损 | 分模块迁移 + feature flag + 集成测试 |
| `community-store` 数据适配复杂 | 高 | 社区回归风险最大 | store 内做兼容适配层，前端页面尽量不动 |
| Auth 统一后角色判断混乱 | 中高 | 登录与权限异常 | middleware 与 role 查询单独做验证清单 |
| Metro Monorepo 解析踩坑 | 中 | App 无法启动 | 在 P0a 早期解决，必要时简化包依赖关系 |
| 图片上传 OOM 或超时 | 中 | 发帖失败、崩溃 | 严控压缩规格，逐张上传，支持恢复 |
| GPS 漂移 | 中 | 找店结果不可信 | B 端坐标校准 + 客户端降级选区 |
| 冷启动内容不足 | 高 | MVP 价值无法验证 | 上线前准备足量商户、酒款、帖子种子数据 |
| 酒类合规或审核压力 | 中 | 上架与运营风险 | 提前补合规文本，审核后台前置上线 |

### 回退规则

1. Auth、各 store 必须支持按模块回退。
2. legacy 逻辑文件可保留为只读回退资产，不立即删除。
3. 回退不能影响其他已迁移模块。
4. 高风险模块上线前必须做“切到新逻辑”和“切回旧逻辑”双向验证。
