# Your Wine Book

## 香港酒品垂直社区 App

## 详细方案解释与执行文档

---

## 0. 文档定位

本文档不是概念性提案，而是一份面向落地执行的项目文档。它解决四件事：

1. 解释为什么这个产品要这样做，而不是那样做。
2. 定义 MVP 的真实边界，避免团队在开发过程中不断加功能。
3. 把技术架构、数据模型、权限边界、客户端链路、后台链路、上线链路讲清楚。
4. 把执行路径拆解成可以分派、排期、验收的小任务。

本文档假设目标是做一款聚焦香港市场的酒品垂直社区 App，先验证两个最核心的闭环：

1. 找店闭环：定位 -> 找附近酒商 -> 收藏 -> 外部导航到店
2. 社区闭环：看内容 -> 发图文酒评 -> 点赞评论互动 -> 再次回访

---

## 1. 项目定义

### 1.1 项目愿景

Your Wine Book 的长期目标不是单纯做“酒商黄页”，也不是一开始就做“酒类电商”，而是做一个以香港本地酒品消费场景为核心的垂直社区。它连接三类要素：

1. 酒商与门店
2. 酒款与内容
3. 用户口碑与线下消费场景

这个产品如果成立，价值不是“信息展示”，而是“交易前决策 + 到店转化 + 社区黏性”三者同时发生。

### 1.2 为什么要收敛成 MVP

如果一开始同时做地图、商品、活动、社区、推荐、私信、通知、视频、直播、会员权益，团队会同时面对以下风险：

1. 前端链路过长，任何一个点卡住都会拖垮上线节奏。
2. 权限和审核模型复杂化，Supabase RLS 很容易失控。
3. App 首版很难达到稳定性要求，尤其是图片上传、定位和列表性能。
4. 冷启动阶段没有足够内容和商家，复杂功能不会创造价值。

因此，本项目必须坚持一个原则：

`先验证高频核心闭环，再决定是否值得做高阶功能。`

### 1.3 MVP 目标

MVP 阶段必须达成以下结果：

1. 用户可以在 App 中快速找到附近酒商。
2. 用户可以收藏门店，并通过外部地图导航到店。
3. 用户可以浏览社区内容，并发图文酒评。
4. 用户可以进行点赞、评论、举报、拉黑等基础社交与风控操作。
5. 商户和管理员可以在后台维护基础数据、处理违规内容。
6. 团队可以清晰观测关键漏斗数据，判断是否继续扩张。

### 1.4 MVP 非目标

以下内容明确不进入 MVP 首发范围：

1. 电商交易闭环
2. 站内 IM 私信
3. 短视频与直播
4. App 内地图路线规划
5. 瀑布流 Feed
6. 智能推荐排序
7. 全量商户开放自助入驻
8. 酒类复杂库存与订单系统

## 2. 用户角色与权限边界

### 2.1 角色定义

#### C 端消费者

能力范围：

1. 注册/登录
2. 浏览门店与帖子
3. 收藏门店与帖子
4. 发帖、点赞、评论
5. 举报帖子、评论、用户
6. 拉黑其他用户

#### 商户员工

能力范围：

1. 登录商户后台
2. 维护商户主体与门店信息
3. 维护营业时间与联系方式
4. 拖拽修正门店坐标
5. 以官方身份发布帖子
6. 管理自家展示酒款

#### 管理员

能力范围：

1. 审核举报
2. 隐藏或下架违规内容
3. 封禁账号
4. 管理商户主体状态
5. 查看系统指标与风控队列

### 2.2 权限原则

必须避免把“用户身份”和“操作权限”揉成一个字段。

正确做法：

1. `profiles` 只存用户基础档案。
2. `merchant_staff` 定义某个用户是否属于某个商户，以及其权限级别。
3. 管理员权限单独控制，不依赖客户端传参。
4. 敏感写操作通过服务端接口或 Edge Function 执行，RLS 只做底线约束。

### 2.3 官方身份与个人身份的区别

这是产品与审计设计里的关键点。

同一个人可能同时具有两个身份：

1. 个人身份：代表自己发酒评
2. 官方身份：代表某商户发布官方动态

因此帖子模型必须显式区分：

1. `author_profile_id`：实际操作者是谁
2. `acting_merchant_id`：是否代表商户发布
3. `is_official`：是否以官方身份发出

不能仅靠 `merchant_id is not null` 推断官方身份，否则后续审计、权限、展示都容易混乱。

### 3.2 冷启动范围

MVP 不应理解为“功能做完即可上线”，而应理解为“功能 + 初始数据 + 初始供给 + 内测策略”同时具备。

最低冷启动建议：

1. 30-50 家门店数据，且坐标已人工校准
2. 200-500 个基础酒款数据
3. 100-200 条内容种子
4. 20-50 名测试用户
5. 5-10 家愿意配合测试的商户

首发建议：

1. UI 默认繁体中文
2. 关键系统文案同步提供英文
3. 用户内容不自动翻译
4. 商户后台支持繁中优先，英文可作为后续增强

## 4. 核心产品逻辑解释

### 4.1 找店闭环为什么成立

香港本地酒品消费具有明显的线下场景属性，用户真正需要的是：

1. 附近哪里可以买
2. 哪家更值得去
3. 门店是否还在营业
4. 导航是否方便

所以找店闭环的核心不是“地图花哨”，而是：

1. 定位准
2. 列表快
3. 信息全
4. 到店动作短

MVP 不应在第一版就投入 App 内路径规划，而应直接拉起 Apple Maps / Google Maps，缩短路径。

### 4.2 社区闭环为什么成立

酒类消费天然适合“内容驱动决策”，但首版社区必须克制。

要避免一开始做复杂分发机制，因为：

1. 冷启动内容不足时，推荐算法没有意义
2. 排序逻辑一复杂，分页、缓存、回流、调试成本都指数上升
3. 移动端首版最需要的是发布稳定、互动及时、违规可控

因此 MVP 社区只保留：

1. 时间倒序单列 Feed
2. 图文帖子
3. 点赞评论
4. 举报拉黑

### 4.3 为什么审核必须前置

只要允许 UGC，审核就不是优化项，而是上线门槛。原因很直接：

1. App Store / Google Play 对 UGC 应用有明确审核要求
2. 酒类社区极易出现广告、引流、售卖争议、色情擦边
3. 如果首版没有举报和下架机制，团队将无法控制风险

所以审核必须从 Phase 0 就存在，而不能等用户多了再补。

## 5. 技术架构方案

### 5.1 总体架构

采用 `pnpm workspace + Turborepo` 的 Monorepo 结构，包含三个层次：

1. `apps/mobile`：Expo / React Native 消费者 App
2. `apps/web`：Next.js 商户后台 + 管理后台
3. `packages/*`：共享纯逻辑包

### 5.2 为什么选 `pnpm workspace + Turborepo`

原因如下：

1. 当前项目只有移动端、Web 端和少量共享包，Turborepo 的任务编排和缓存能力已经足够。
2. `pnpm` 对 workspace 依赖隔离更清晰，适合控制包边界。
3. 不需要为了图谱分析、生成器和大型组织治理过早引入更重的 monorepo 体系。
4. Expo 与 Next.js 在 Monorepo 下的主要难点不是“构建能力”，而是 “metro 兼容 + 依赖边界”，这部分通过约定和配置解决即可。

### 5.3 目录建议

```text
apps/
  mobile/
  web/
packages/
  domain/
  query-keys/
  supabase-types/
  ui-tokens/
  eslint-config/
  tsconfig/
supabase/
  migrations/
  seeds/
  functions/
docs/
  product/
  engineering/
```

### 5.4 共享层边界

共享层只放“纯逻辑”，不放“运行时耦合代码”。

允许共享：

1. TypeScript 类型定义
2. Zod 校验规则
3. 常量字典
4. 业务枚举
5. React Query keys
6. 通用数据格式转换函数

不允许共享：

1. fetch 封装
2. storage adapter
3. SecureStore / localStorage 调用
4. Web Cookie 或服务端 session 逻辑
5. React Native 专属 hooks
6. Next.js server action 逻辑

### 5.5 客户端状态设计

#### React Query

负责：

1. 服务端数据获取
2. 分页缓存
3. 乐观更新
4. 失败重试
5. 弱网回显

#### Zustand

只负责：

1. 筛选项
2. UI 弹窗状态
3. 本地发帖草稿状态
4. 临时表单状态

#### Auth Provider

单独负责：

1. session 初始化
2. token refresh
3. 登录态恢复
4. 当前用户基础信息拉取

Auth 不放 Zustand，避免 session 生命周期和本地 UI 状态混在一起。

### 5.6 网络与离线策略

香港地铁、电梯、商场等场景下弱网很常见，MVP 必须一开始就定义策略：

1. 读场景使用 React Query 持久化缓存
2. 最近一次成功加载的数据可以离线回显
3. 列表页断网时不白屏
4. 写场景区分“可重试”和“必须前台确认”

具体策略：

1. Feed、门店列表、门店详情、帖子详情可缓存回显
2. 发帖上传必须显式展示状态
3. 失败的帖子上传进入草稿箱
4. 点赞/评论操作使用幂等设计

### 5.7 埋点与监控

必须在 Phase 0 前置：

1. Sentry：崩溃监控、错误堆栈
2. PostHog：漏斗、页面、行为埋点
3. 日志分层：客户端日志、服务端日志、审核操作日志

### 5.8 Deep Link 与分享

必须前置：

1. App URL Scheme
2. Universal Link / App Link
3. 帖子详情分享链接
4. 门店详情分享链接
5. 邀请码或灰度用户落地页

原因是这类配置涉及 iOS Associated Domains 和 Android 原生配置，后补成本高。

### 5.9 国际化策略

首版即接入 i18n 框架，不要在 MVP 之后再补。

原因：

1. UI 文案会快速分散到 App 和后台多个位置
2. 如果后补，多数页面都要返工
3. 香港本地产品天然需要处理中英文共存

建议：

1. UI 文案通过 key 管理
2. 默认繁中
3. 系统错误、登录提示、审核提示同步支持英文

### 6.3 状态字段建议

尽量统一状态语义，避免每张表自由发挥。

示例：

1. `draft`
2. `published`
3. `hidden`
4. `deleted`
5. `banned`
6. `pending_review`

### 6.4 索引建议

必须有：

1. `merchant_locations.location` GIST 空间索引
2. `posts(created_at desc, id desc)` 复合索引
3. `post_comments(post_id, created_at desc)`
4. `post_likes(post_id, profile_id)` 唯一索引
5. `merchant_bookmarks(profile_id, merchant_id)` 唯一索引
6. `post_bookmarks(profile_id, post_id)` 唯一索引
7. `merchant_staff(merchant_id, profile_id)`
8. `reports(target_type, target_id)`
9. `moderation_cases(status, risk_level, created_at)`

### 6.5 RLS 原则

RLS 目标不是“把业务逻辑全塞进数据库”，而是保证底线安全。

建议原则：

1. 普通用户只能修改自己的资料、自己的帖子、自己的评论、自己的收藏、自己的举报。
2. 商户员工只能管理与自己关联的商户及门店。
3. 审核与封禁动作不开放给普通客户端。
4. 任何官方发帖都必须校验操作者是否属于对应商户。

## 7. 关键链路设计

### 7.1 定位与门店检索链路

#### 目标

让用户在最短路径内看到附近门店，并且结果可信。

#### 流程

1. App 首次进入请求定位权限
2. 用户授权成功后获取坐标
3. 调用附近门店接口
4. 服务端先用 `ST_DWithin` 按半径过滤，再按距离排序
5. 如果 5km 无结果，则扩圈到 10km
6. 如果 10km 仍无结果，则扩圈到 20km
7. 若仍无结果，返回全港精选
8. 客户端优先展示已收藏的附近门店，再展示全部附近门店

#### 注意点

1. 香港 GPS 漂移明显，后台必须支持手动拖图钉纠偏。
2. 门店列表必须展示营业状态、距离、地址摘要。
3. 用户拒绝定位时，要允许手动选择区域。

### 7.2 发帖与上传链路

#### 核心要求

上传链路必须可控、可重试、可审计，不能让客户端直接写最终媒体记录。

#### 推荐流程

1. 用户选图
2. 客户端立即生成本地低清预览
3. 客户端压缩图片
4. 客户端调用 `create_upload_intents`
5. 服务端为每张图生成可上传路径和授权
6. 客户端逐张上传到 Storage
7. 上传完成后，客户端提交 `finalize_post`
8. 服务端校验上传对象确实属于该用户、状态完整后，写入 `posts + post_media`
9. 返回正式帖子结果

#### 为什么不能直接客户端写 `post_media`

因为会有以下风险：

1. 客户端伪造他人 URL
2. 上传未完成却写入成功记录
3. 无法做统一审计
4. 媒体尺寸、文件类型、大小难以统一校验

### 7.3 Feed 分页链路

MVP 阶段必须坚持时间倒序单列流，不做复杂混排。

原因：

1. 时间倒序的游标分页最稳定
2. `(created_at, id)` 可以天然做游标
3. 便于本地缓存和下拉刷新
4. 能降低“数据重复/漏数据/回刷跳动”等问题

### 7.4 点赞、评论、举报链路

#### 点赞

要求：

1. 幂等
2. 支持乐观更新
3. 弱网重试后不重复

实现建议：

1. 服务端以联合唯一键保障幂等
2. 客户端使用 mutation + rollback

#### 评论

要求：

1. 发表评论后立即可见
2. 违规评论可快速隐藏
3. 支持分页加载

#### 举报

要求：

1. App 端任何帖子、评论、用户主页都可发起举报
2. 举报后即时写入 `reports`
3. 后台生成或关联 `moderation_cases`
4. 管理员可直接处理

### 7.5 拉黑链路

拉黑不是附加功能，而是合规与体验需求。

拉黑后至少要影响：

1. 目标用户内容在当前用户 Feed 中隐藏
2. 对方主页不可达或降级展示
3. 评论区可按产品规则隐藏或折叠

## 9. 实施原则

### 9.1 先建底座，再做页面

优先顺序必须是：

1. Schema 与权限
2. 上传与审核
3. 核心 API
4. 页面开发
5. 指标与发布

不能反过来，否则 UI 会不断返工。

### 9.2 先直线闭环，再做体验优化

例如：

1. 先能发帖成功，再做高级草稿箱
2. 先能找到店，再做品牌地图
3. 先能审核下架，再做自动风控

### 9.3 所有关键动作都要定义观测口径

没有数据定义的功能，上线后无法判断成功与否。

例如：

1. 定位权限授权率
2. 导航点击率
3. 发帖成功率
4. 上传失败率
5. 审核处理时长

## 14. 指标与验收标准

### 14.1 核心业务指标

1. 定位授权率
2. 附近门店曝光量
3. 门店收藏率
4. 外部导航点击率
5. 发帖点击率
6. 发帖成功率
7. 点赞率
8. 评论率
9. 举报处理时长
10. D1 / D7 留存

### 14.2 性能与稳定性指标

1. 4G 网络下，点击图标到“附近门店首屏可交互” `TTI <= 3s`
2. 热启动回到首页时间 `<= 1s`
3. 单次最多 9 张图发帖成功率 `> 98%`
4. 审核隐藏内容后 60 秒内客户端不可见
5. 崩溃率维持在可接受范围

### 14.3 功能验收标准

#### 找店闭环验收

1. 用户授权定位后能看到附近门店

3. 可收藏门店
4. 可拉起外部地图导航

1. 用户能成功发布图文帖子
2. 帖子发布失败时不会丢失内容
3. 点赞评论流畅可用
4. 举报与拉黑可用
5. 管理员可后台快速下架违规内容

## 16. 风险清单与应对

### 风险 1：定位不准导致用户不信任

应对：

1. 商户后台拖拽坐标微调
2. 结果页显示距离时不过度强调极小数值
3. 提供地址与导航双重确认

### 风险 2：图片上传不稳定导致首版口碑差

应对：

1. 客户端压缩
2. 上传进度展示
3. 草稿箱
4. finalize 服务端兜底

### 风险 3：RLS 复杂度过高拖慢进度

应对：

1. 只把底线权限放进 RLS
2. 复杂写操作走服务端接口
3. 一开始就写权限测试

### 风险 4：冷启动内容不足导致社区空转

应对：

1. 运营种子内容前置
2. 邀请商户员工参与官方发帖
3. 邀请制内测，不急着公开放量

### 风险 5：审核工作量超预期

应对：

1. 首版先做人工审核后台
2. 设计清晰的举报理由分类
3. 把一键隐藏放在最高优先级

## 18. 上线后的 2 周观察重点

上线后不要立刻扩需求，先盯下面几项：

1. 定位授权率是否达标
2. 门店列表是否真的被点击与收藏
3. 导航点击率是否有意义
4. 发帖成功率是否足够高
5. 用户是否会在发帖后获得互动
6. 举报量和审核压力是否可控
7. D1 留存是否足够支撑继续投入

如果这几项不成立，Phase 2 应暂停，先修闭环和冷启动。

---

## 19. 结论

这个项目的成功关键不在于“功能多”，而在于首版能否同时做到三件事：

1. 找店真的有用
2. 发帖真的稳定
3. 审核真的可控

所以执行上要始终坚持以下顺序：

1. 先底座
2. 再闭环
3. 再稳定性
4. 最后才是增强体验

如果团队严格按本文档推进，那么在 8-10 周内做出一个能内测、能灰度、能验证价值的 MVP 是现实可行的。
