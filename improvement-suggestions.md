# Your Wine Book — 系統完善建議

> 針對用戶系統、酒商管理、酒款上架管理三個方面的改進建議。

---

## 一、用戶系統完善

### 1.1 當前狀態

- 用戶數據存在 `data/users.json`，每次讀寫都操作文件
- 密碼明文存儲（`password` 字段直接存原始密碼）
- 無密碼找回流程
- 書籤（收藏）功能已實現，存在用戶 JSON 中
- 無用戶頭像、無社交登入

### 1.2 建議改進

#### A. 密碼加密（優先級：高）

目前密碼明文存儲，這是安全隱患。建議：
- 使用 `bcrypt` 對密碼做哈希，註冊和改密碼時存哈希值
- 登入時用 `bcrypt.compare()` 驗證
- 遷移現有帳號：寫一次性腳本把現有明文密碼轉為哈希

#### B. 降價提醒（優先級：中）

用戶收藏的酒款價格下降時通知。實現思路：
- 在 `users.json` 中為每支收藏酒款記錄「收藏時的最低價」
- 新增一個定時任務（或手動觸發的 API），對比當前最低價和收藏時的價格
- 價格下降時標記提醒，用戶下次登入時在 Navbar 顯示通知小紅點
- 後期可擴展為 Email 通知

#### C. 用戶偏好與個性化（優先級：低）

- 記錄用戶的瀏覽/收藏偏好（常看的酒類型、價位段、產區）
- 首頁精選酒款根據偏好排序
- 搜索頁默認排序可按個人偏好調整

## 二、酒商管理

### 2.1 當前狀態

- 酒商帳號存在 `data/merchants.json`（含密碼、狀態、聯繫方式）
- 管理員可以在後台創建帳號、啟用/停用
- 入駐申請有獨立工作流（待處理→已聯繫→已批准/已拒絕）
- 申請批准後不會自動創建酒商帳號，需要管理員手動創建
- 密碼同樣是明文存儲

### 2.2 建議改進

#### A. 申請→開戶自動化（優先級：高）

目前管理員批准申請後，還要手動去「酒商帳號」頁面創建帳號，流程斷裂。建議：
- 批准申請時自動彈出「創建帳號」表單，預填公司名、Email 等信息
- 或者直接一鍵：批准 + 創建帳號 + 生成臨時密碼
- 創建成功後顯示臨時密碼供管理員複製發送給酒商（後期可改為自動發 Email）

#### B. 酒商資料完善（優先級：中）

目前酒商的展示信息比較單薄。建議增加：
- **營業時間**：有實體店的酒商可標註
- **配送範圍/運費**：幫用戶在比價時考慮運費因素
- **退換政策**：增加信任度
- **特色標籤**：如「專注法國酒」「免運費」「有實體店」「24小時出貨」
- 這些信息在酒商詳情頁和比價表中展示

#### C. 酒商數據面板增強（優先級：中）

目前酒商後台的數據比較基礎。建議增加：
- **競爭對手比價趨勢**：某支酒，你的價格 vs 市場平均 vs 最低價的歷史走勢
- **曝光 vs 點擊漏斗**：你的酒款在搜索結果中出現了多少次 → 被點進詳情頁多少次 → 被點擊購買連結多少次
- **價格建議**：基於平台數據，提示「你的 XX 酒款比市場最低高 15%，調價後預估點擊增加 XX%」

## 三、酒款上架管理

### 3.1 當前狀態

- 新增酒款表單已有，但 Demo 模式不入庫
- 無審核流程（表單顯示「提交審核」但實際沒有審核機制）
- 無批量導入功能
- 酒款數據和比價數據分離（`mock-data.ts` 中的 `wines` 和 `winePrices`）
- 無法編輯已上架酒款、無法修改價格

### 3.2 建議改進

#### A. 酒款上架真實流程（優先級：高）

建立完整的從提交到上架的流程：

```
酒商提交酒款
    ↓
平台待審核列表（管理員後台新增頁面）
    ↓ 管理員審核
通過 → 酒款上架（寫入 wines + wine_prices）
拒絕 → 通知酒商原因
```

具體改動：
- `data/` 下新增 `wine-submissions.json` 存放待審核酒款
- 管理員後台新增「酒款審核」頁面
- 審核通過後：
  - 如果是平台已有酒款（匹配 slug）→ 只新增這個酒商的價格記錄
  - 如果是全新酒款 → 創建酒款 + 價格記錄

#### B. 酒款價格編輯（優先級：高）

酒商最常見的操作是更新價格和購買連結。建議：
- 在酒款管理列表中，每支酒旁邊加「編輯價格」按鈕
- 點擊後彈出小表單：新價格 + 新購買連結
- 提交後即時生效（不需要審核，因為價格是酒商自己的數據）
- 記錄價格變更歷史（方便後期做價格走勢圖）

#### C. CSV 批量導入（優先級：中）

新酒商入駐時通常有幾十甚至上百支酒要上架，逐個填寫不現實。建議：
- 提供 CSV 模板下載（列：酒名、類型、年份、葡萄品種、產區、價格HKD、購買連結）
- 上傳後前端解析預覽（顯示表格讓酒商確認）
- 確認後批量提交到待審核列表
- 支持增量導入（已有酒款只更新價格）

#### D. 酒款匹配與去重（優先級：中）

多個酒商上架同一支酒時，需要確保是同一條酒款記錄下的不同價格，而不是創建重複酒款。建議：
- 酒商提交時，根據酒名 + 年份做模糊匹配
- 匹配到已有酒款 → 提示「平台已有此酒款，確認新增你的價格？」
- 未匹配到 → 創建新酒款條目
- 管理員審核時也可手動合併重複酒款

#### E. 酒款下架/標記缺貨（優先級：中）

酒商斷貨時需要標記：
- 在酒款管理列表加「標記缺貨」開關
- 缺貨酒款在前台比價表中灰顯，標記「暫時缺貨」
- 不從比價表中刪除（保留歷史價格參考）
- 可設置「到貨通知」，酒商重新上架時通知收藏了該酒的用戶

## 四、建議開發優先級

### 第一階段（建議先做）

| 項目 | 原因 |
|------|------|
| 酒款上架真實流程 | 目前「新增酒款」是空的，核心業務流程缺失 |
| 酒款價格編輯 | 酒商最高頻操作，沒有這個功能後台等於沒用 |
| 申請→開戶自動化 | 減少管理員手動操作，完善入駐流程 |
| 密碼加密 | 安全底線 |

### 第二階段

| 項目 | 原因 |
|------|------|
| CSV 批量導入 | 新酒商入駐效率 |
| 酒款匹配與去重 | 數據質量保障 |
| 酒款下架/標記缺貨 | 信息準確性 |
| 酒商資料完善 | 提升用戶決策信息量 |

### 第三階段

| 項目 | 原因 |
|------|------|
| 降價提醒 | 用戶留存與回訪 |
| 酒商數據面板增強 | 酒商續費/付費的核心價值 |
| 用戶偏好與個性化 | 提升轉化率 |
